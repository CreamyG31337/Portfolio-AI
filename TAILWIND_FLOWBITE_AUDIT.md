### 1. `theme.css` Overrides Tailwind Utilities
**Issue:** `web_dashboard/static/css/theme.css` uses `!important` to override core Tailwind classes (e.g., `.bg-white`, `.text-gray-900`) to apply CSS variables for theming.
**Why:** This breaks the "utility-first" contract. Developers expect `.bg-white` to be white, not a variable that might be dark gray. It makes the codebase fragile and hard to reason about. It forces the framework to work against itself.
**Suggestion:** Move these color definitions into `web_dashboard/tailwind.config.js` under `theme.extend.colors`. Map the utilities to the CSS variables there (e.g., `colors: { white: 'var(--bg-primary)' }` if that's the intent, or better, use semantic names like `bg-surface`). Remove the `!important` overrides in CSS.
**Scope:** Systemic

### 2. Inline Styles in `base.html`
**Issue:** `web_dashboard/templates/base.html` contains a `<style>` block defining `.gradient-bg`, `.sidebar-text`, and transitions.
**Why:** Custom CSS prevents reuse and consistency. Tailwind provides utilities for gradients (`bg-gradient-to-r`) and transitions (`transition-all`).
**Suggestion:** Replace `.gradient-bg` with `bg-gradient-to-br from-indigo-500 to-purple-600` (or similar). Move complex custom animations to `input.css` `@layer components` if they cannot be composed.
**Scope:** Local (Base Template)

### 3. Duplicate Sidebar Logic
**Issue:** `web_dashboard/templates/components/_scripts_content.html` contains ~150 lines of custom JavaScript (`collapseSidebar`, `expandSidebar`) to manage the sidebar state.
**Why:** This logic reinvents the wheel. Flowbite's `Drawer` component (already included via CDN) handles showing/hiding sidebars, including mobile overlays and transitions. The custom code creates potential conflicts and maintenance burden.
**Suggestion:** Refactor to use Flowbite's [Sidebar/Drawer component](https://flowbite.com/docs/components/sidebar/) fully. Use `data-drawer-target` attributes for toggling. For desktop collapsing, standard Tailwind state classes (e.g., peer-checked or simple class toggling) are cleaner than manual inline style manipulation.
**Scope:** Reusable Component (Sidebar)

### 4. Inline JavaScript in Templates
**Issue:** `base.html` and `_scripts_content.html` contain significant inline JavaScript for UI behavior (header auto-hide, theme polling).
**Why:** Inline scripts are harder to lint, test, and cache. They clutter the HTML structure.
**Suggestion:** Extract this logic into `web_dashboard/src/js/ui.ts` or similar, compile it, and include it.
**Scope:** Systemic
